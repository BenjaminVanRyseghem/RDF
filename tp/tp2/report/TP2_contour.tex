\section{Codage d'un contour}

\subsection{Comment sont codés les points du contour dans la variable cont?}
Les points du contour sont stockés dans la variable sous forme d'une liste de nombres complexes. La valeur réel correspondant à $x$ et la valeur imaginaire à $y$.

\subsection{En ne conservant qu'un point sur 4, puis un point sur 8, constituer deux autres contours approchant la forme circulaire et les afficher respectivement en bleu et en vert dans la même fenêtre}

Voici le code correspondant:

\begin{verbatim}
nom1 = "cercle-80.txt";
cont1 = rdfChargeFichierContour (nom1);

cont2 = cont1(1:4:$);
cont3 = cont1(1:8:$);
rdfAfficheContour(cont1, 2, "r");
rdfAfficheContour(cont2, 2, "g");
rdfAfficheContour(cont3, 2, "b");
\end{verbatim}

L'exécution du code précédent fournit la Fig.~\ref{fig:1_sur_4__1_sur_8}, sur laquelle apparaissent trois contours. Il y a d'abord le contour initial d'une forme circulaire représentée en rouge.

Il s'y trouve également le contour vert résultant du codage du contour original en ne conservant qu'un point sur quatre. 

Enfin, il y a le contour bleu représentant le codage du contour original en ne gardant cette fois qu'un point sur huit.

\section{Filtrage des descripteurs de Fourier}

Voici la fonction rdfAnnuleDescFourier:

\begin{verbatim}
function vector = rdfAnnuleDescFourier(desc,ratio)
    size = size(desc,1);
    
    if ratio == 1 then
        vector = desc
    else 
        if ratio == 0 then
            vector = zeros(1,size)
        else
            index = int(ratio*size)
            vector = [desc(1:index)' zeros(1,size-index) ]'
         end
    end
    
endfunction
\end{verbatim}


Lorsque nous réduisons le ratio il se trouve qu'en effet la forme se simplifie (\emph{c.f.}~Fig.~\ref{fig:reduction_fourrier})

\section{Réduction d'une chaîne de contour}

Voici le code que nous avons mis pour rdfAlgorithmeCorde:
\begin{verbatim}

    // Distances des autres points a la droite debut<->fin
    // le vecteur d, de meme taille que le vecteur
    // 'cont' doit contenir les distances

    // si c'est une droite verticale
    if real (debut) == real (fin)  then
        d = abs(real(cont) - real(debut))
    else	
        // on recupere le a et le b de ax + b
        a = (imag(fin) - imag(debut))/(real(fin) - real(debut))
        b = imag(fin) - a * real(fin)
        		
        // si c'est une droite horizontale
        if a == 0 then
            d = abs(imag(cont) - b)
        else
            d = abs(a*real(cont) - imag(cont) + b) ./ (sqrt(1+a^2))
        end
    end
\end{verbatim}



Et on observe bien que si on augmente \verb?dmax? on réduit bien le nombre de point du contour (\emph{c.f.}~Fig.~\ref{fig:reduction_chaine_cont})

\section{Comparaison des deux approches}

Après avoir fait quelques tests nous nous sommes rendu compte que pour des formes connexes (triangles, carres, cercles) l'algorithme des cordes est mieux car beaucoup plus rapide que celui des descripteurs de Fourier. Cependant, dans les deux cas l'approximation est correcte (\emph{c.f.}~Fig.~\ref{fig:comp_fourier}).

\begin{figure}[ht]
	\begin{center}
		\includegraphics[width=12cm]{cmp_fourier_normale.png}
	\end{center}
	\caption{rouge: contours initial \_\_ vert: Fourier avec ratio=0.95 \_\_ bleu: Cordes avec dmax=1}
	\label{fig:comp_fourier}
\end{figure}

Mais pour ce qui est des formes non connexes (patatoïde) l'algorithme de Fourier donne un meilleur aperçu de la forme d'origine. On observe sur la figure Fig.~\ref{fig:comp_fourier} que l'algorithme des cordes fait des segment qui sont moins représentatif de la forme de base.

En conclusion nous dirons que l'algorithme des cordes est moins efficace pour faire de la reconnaissance de formes complexes comme un visage ou une patate (images réelles) contrairement à l'algorithme de Fourier. Par contre il est beaucoup plus rapide pour des formes géométriques (images générées) comme un triangle ou un carré.
